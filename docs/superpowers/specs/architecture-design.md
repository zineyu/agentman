# Agent任务管理系统 - 架构设计方案

## 1. 架构总览

采用**"飞书Base直连"**模式：飞书多维表格作为唯一数据层，Agent Daemon直接通过飞书OpenAPI读写数据，无中间服务层。

```
┌─────────────────────────────────────────────────────────────┐
│                     飞书多维表格 (核心数据层)                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ 任务主表  │  │ 执行记录表│  │ 运行时表  │  │ 技能表    │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ 依赖关系表│  │ 评论表    │  │ 工时表    │  │ 标签表    │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
│                    ↑ Base API / Webhook                    │
└─────────────────────────────────────────────────────────────┘
                    │              │
           ┌───────┘              └───────┐
           │                              │
    ┌──────┴──────┐               ┌──────┴──────┐
    │  飞书工作流  │               │  Agent      │
    │  (自动化)   │               │  Daemon     │
    └─────────────┘               │  (Rust)     │
                                  └──────┬──────┘
                                         │
                              ┌──────────┼──────────┐
                              │          │          │
                         ┌────┴───┐ ┌───┴───┐ ┌───┴───┐
                         │Claude  │ │Codex  │ │Open   │
                         │Code    │ │       │ │Code   │
                         └────────┘ └───────┘ └───────┘
```

### 核心组件

| 组件 | 技术 | 职责 |
|------|------|------|
| **飞书多维表格** | 飞书Base | 数据持久化、任务状态管理、权限控制、协作界面 |
| **飞书工作流** | 飞书自动化 | 状态流转触发、条件通知、Webhook调用 |
| **Agent Daemon** | Rust | 任务监听、Agent CLI检测、任务执行、结果回写 |
| **Agent CLI** | 各厂商CLI | 实际执行代码任务的Agent（Claude Code、Codex、OpenCode等） |

## 2. 数据模型设计（飞书Base表结构）

### 2.1 任务主表（Tasks）

| 字段名 | 字段类型 | 说明 | 约束 |
|--------|----------|------|------|
| 任务编号 | 自动编号 | 唯一标识，如 TASK-001 | 系统自动生成 |
| 任务标题 | 文本 | 简短描述任务内容 | 必填 |
| 任务描述 | 多行文本 | 详细需求描述 | 可空 |
| 执行者类型 | 单选 | human / agent | 必填，决定后续行为 |
| 执行者 | 人员/文本 | human填人员，agent填agent标识 | 必填 |
| 任务状态 | 单选 | 待办/进行中/待审核/已完成/已取消 | 必填，有工作流约束 |
| 优先级 | 单选 | P0/P1/P2/P3 | 默认P2 |
| 标签 | 多选 | 关联标签表 | 可空 |
| 父任务 | 关联 | 自关联，支持任务树 | 可空 |
| 预计工时 | 数字 | 小时数 | 可空 |
| 实际工时 | 数字 | 小时数（工时表汇总） | 公式字段，只读 |
| 开始时间 | 日期时间 | 任务开始执行时间 | 可空 |
| 截止时间 | 日期时间 | 期望完成时间 | 可空 |
| 完成时间 | 日期时间 | 实际完成时间 | 可空 |
| 创建人 | 人员 | 系统自动 | 只读 |
| 创建时间 | 日期时间 | 系统自动 | 只读 |
| 最后更新 | 日期时间 | 系统自动 | 只读 |
| Agent类型 | 单选 | claude-code / codex / opencode / cursor / ... | agent任务必填 |
| 工作目录 | 文本 | Agent执行时的本地路径 | agent任务必填 |
| 前置任务 | 关联 | 指向任务主表，依赖的任务 | 可空 |
| 审核人 | 人员 | 待审核状态的审核责任人 | 可空 |
| 审核意见 | 多行文本 | 审核反馈 | 可空 |
| 审核驳回理由 | 多行文本 | 审核驳回时的具体原因 | 可空，驳回时必填 |

| 催办次数 | 数字 | 催办统计 | 默认0，agent任务不增加 |
| 最后催办时间 | 日期时间 | 催办时间戳 | 可空 |
| 分配的运行时 | 关联 | 指向运行时表，预分配执行Daemon | agent任务必填 |

## 3. 飞书工作流设计

### 3.1 状态流转自动化

**触发器**：任务主表 - 记录更新（任务状态字段变更）

```
状态流转规则：

[待办] → [进行中]
  触发：执行者点击"开始"
  动作：记录开始时间

[进行中] → [待审核]  
  触发：Agent任务自动完成 或 人类提交完成
  条件：执行者类型 = agent
  动作：设置审核人 = 创建人，发送审核通知

[进行中] → [已完成]
  触发：人类任务提交完成
  条件：执行者类型 = human
  动作：记录完成时间

[待审核] → [已完成]
  触发：审核人点击"通过"
  动作：记录完成时间，通知执行者

[待审核] → [进行中]
  触发：审核人点击"驳回"
  条件：重试次数 < 3
  动作：
    1. 将审核驳回理由追加到评论
    2. 重试次数 + 1
    3. 更新任务状态为"进行中"
    4. Daemon重新执行（使用驳回理由作为上下文）

[任意] → [已取消]
  触发：创建人点击"取消"
  动作：记录取消时间
```

### 3.2 催办工作流（仅人类任务）

**触发器**：定时触发（每天10:00）

```
条件：
  - 任务状态 = 待办 或 进行中
  - 执行者类型 = human
  - 截止时间 < 当前时间 + 24小时（即将到期）
  - 催办次数 < 3

动作：
  1. 催办次数 + 1
  2. 更新最后催办时间
  3. 发送飞书消息给执行者
  4. 添加系统评论："系统自动催办（第X次）"
```

### 3.3 Agent任务分配触发

**触发器**：任务主表 - 记录新增（执行者类型 = agent）

```
动作：
  1. 验证关联的运行时是否在线（通过运行时表查询）
  2. 记录系统评论："Agent任务已分配，等待执行"
  3. 如运行时离线，通知创建人并标记为"待分配"
```

> **注**：采用任务预分配机制。创建Agent任务时必须指定"分配的运行时"字段，
> 只有该运行时对应的Daemon会执行此任务，避免多Daemon竞争。

## 4. Agent Daemon设计

### 4.1 技术选型

- **语言**：Rust
- **主要依赖**：
  - `tokio` - 异步运行时
  - `reqwest` - HTTP客户端（调用飞书API）
  - `serde` - 序列化
  - `notify` - 文件系统监听（可选）
  - `tokio-tungstenite` - WebSocket客户端（如需要实时推送）
  - `which` - 检测CLI工具
  - `clap` - CLI参数解析

### 4.2 核心模块

```rust
// 简化的模块结构
agent-daemon/
├── src/
│   ├── main.rs              # 入口，配置加载
│   ├── config.rs            # 配置文件管理
│   ├── base_client.rs       # 飞书Base API客户端
│   ├── task_poller.rs       # 任务轮询器
│   ├── agent_manager.rs     # Agent CLI管理
│   ├── executor.rs          # 任务执行器
│   ├── heartbeat.rs         # 心跳上报
│   └── models.rs            # 数据模型
├── Cargo.toml
└── config.toml              # 配置文件
```

### 4.3 配置项

```toml
# config.toml
[daemon]
# Daemon标识
runtime_id = "auto"  # auto = 自动生成UUID
hostname = "auto"    # auto = 使用系统主机名

[feishu]
# 飞书Base标识
base_token = "bascnxxx"
# 认证方式：复用lark-cli的tenant_access_token
# Daemon启动时自动读取 ~/.lark-cli/config.json 获取凭证

[polling]
# 轮询间隔（秒）
interval = 30
# 同时执行的最大任务数
max_concurrent = 2

[agents]
# 要检测的CLI命令（留空则禁用该Agent）
claude_code = "claude"
codex = "codex"
opencode = "opencode"
cursor = "cursor"

[execution]
# 默认工作目录（仓库clone的根目录）
default_work_dir = "/home/user/projects"
# 超时时间（分钟）
default_timeout = 30
# 日志回写策略
log_batch_size = 10        # 每10条日志批量回写
log_realtime_keywords = ["error", "failed", "success", "commit", "push"]  # 包含这些关键词的日志实时回写

```

### 4.4 工作模式

**模式一：轮询模式（默认）**

```
1. Daemon启动
   ↓
2. 注册运行时（在运行时表创建/更新记录）
   ↓
3. 检测本地可用Agent CLI
   ↓
4. 每30秒轮询任务主表：
   - 查询条件：
     * 执行者类型 = agent
     * 任务状态 = 待办
     * Agent类型 在 Daemon可用的CLI列表中
     * 无未完成的阻塞型依赖任务（前置任务状态=已完成）
   ↓
5. 获取到可执行任务 → 检查前置依赖是否全部完成
   ↓
6. 依赖检查通过 → 更新状态为"进行中"
   ↓
7. 调用对应Agent CLI执行任务
   ↓
8. 监控执行过程，日志回写策略：
   a. 普通日志：缓冲10条后批量回写到执行记录表
   b. 关键词日志（error/failed/success/commit/push）：实时回写
   ↓
9. 执行完成 → 更新任务状态为"待审核"
   ↓
10. 返回步骤4继续轮询
```

### 4.5 Agent CLI调用规范

以Claude Code为例：

```bash
# 在指定工作目录执行
claude -p "<提示词模板>" --allowedTools "Bash,Edit,View" 

# 或者使用非交互模式
claude "<任务描述>" --output-format json
```

不同Agent的调用方式需要适配器模式：

```rust
trait AgentAdapter {
    fn name(&self) -> &str;
    fn is_available(&self) -> bool;
    fn execute(&self, task: &Task) -> Result<ExecutionResult>;
}

struct ClaudeCodeAdapter;
struct CodexAdapter;
struct OpenCodeAdapter;
```

## 6. 权限设计

利用飞书Base原生权限：

| 角色 | 权限 |
|------|------|
| **管理员** | 全部表读写、修改表结构 |
| **项目经理** | 任务主表读写、可分配任务 |
| **开发者** | 任务主表读 + 更新自己负责的任务 |

## 8. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 飞书API限流 | Daemon任务获取延迟 | 增加指数退避重试、本地缓存 |
| Agent CLI执行超时 | 任务挂死 | 设置超时、支持取消信号 |
| 多Daemon竞争同一任务 | 重复执行 | Base记录加乐观锁、唯一约束 |
| 任务依赖死锁 | 任务无法执行 | 依赖检测、循环依赖校验 |
| 飞书工作流延迟 | 状态流转不及时 | 关键路径走Daemon直接更新 |
| 敏感代码泄露 | 安全风险 | 仓库白名单、权限最小化 |

## 9. 部署架构

### 9.1 Agent Daemon部署

```yaml
# docker-compose.yml (示例)
version: '3.8'
services:
  agent-daemon:
    image: agent-daemon:latest
    container_name: agent-daemon
    restart: unless-stopped
    volumes:
      - ./config.toml:/app/config.toml
      - /var/run/docker.sock:/var/run/docker.sock  # 如需Docker内执行Agent
      - ~/projects:/projects  # 代码仓库目录
    environment:
      - FEISHU_APP_ID=${FEISHU_APP_ID}
      - FEISHU_APP_SECRET=${FEISHU_APP_SECRET}
      - BASE_TOKEN=${BASE_TOKEN}
    # 或直接使用systemd服务在宿主机运行
```

### 9.2 多环境支持

- **开发环境**：本地Daemon，指向测试Base
- **测试环境**：CI/CD机器部署Daemon，使用独立Base
- **生产环境**：专用服务器部署，使用生产Base
